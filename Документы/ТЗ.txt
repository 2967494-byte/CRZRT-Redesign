2. ГЛОССАРИЙ (ТЕРМИНЫ И ОПРЕДЕЛЕНИЯ)

2.1. Адаптивность (Responsive Web Design) — комплекс технологических решений (медиа-запросы, гибкие сетки, адаптивные изображения), позволяющий интерфейсу Системы динамически подстраиваться под параметры отображения клиентского устройства (разрешение экрана, плотность пикселей, ориентация). Целевой диапазон адаптивности: от 320px (мобильные устройства) до 2560px (Ultra-HD мониторы).
2.2. Аутентификация (Authentication) — процедура установления подлинности субъекта (пользователя), осуществляемая программными средствами Системы путем проверки предоставленных идентификационных данных (логин, хешированный пароль) на соответствие эталонным значениям, хранящимся в защищенном реестре.
2.3. Авторизация (Authorization) — процесс сопоставления аутентифицированного субъекта с набором разрешенных действий и ресурсов в рамках Системы на основе заданной матрицы доступа (Role-Based Access Control).
2.4. Бэкенд (Backend / Server-side) — совокупность программных компонентов, функционирующих на стороне сервера, реализующих логику обработки бизнес-процессов, взаимодействие с базами данных, управление сессиями пользователей и формирование ответов для клиентской части.
2.5. База данных (Database) — структурированная совокупность данных, организованная для эффективного поиска, хранения и модификации информации. В рамках проекта предполагается использование реляционных (SQL) или документо-ориентированных (JSON-файлы) хранилищ.
2.6. Дамп (Database Dump) — результирующий файл процесса экспорта данных, содержащий инструкции на языке SQL или структуру объектов в формате JSON, позволяющий полностью восстановить состояние информационной системы на момент создания дампа.
2.7. Инфооблок (Information Block) — атомарная единица контента или функциональный виджет, обладающий уникальным идентификатором, набором атрибутов и закрепленным визуальным шаблоном (например, «Лента новостей», «Карточка услуги»).
2.8. Кэширование (Caching) — метод временного депонирования часто запрашиваемых данных в высокопроизводительных областях памяти (ОЗУ, Redis, браузерный кэш) с целью минимизации циклов обращения к основной БД и снижения нагрузки на центральный процессор сервера.
2.9. Мета-теги (Meta Tags) — специализированные элементы языка разметки HTML, не отображаемые непосредственно пользователю, но несущие служебную информацию для поисковых алгоритмов, социальных сетей (Open Graph) и браузеров.
2.10. ПДО (PHP Data Objects, PDO) — расширение языка программирования PHP, представляющее собой легковесный и надежный интерфейс для доступа к СУБД. Использование PDO является обязательным для обеспечения защиты от атак типа «SQL-инъекция» через механизм подготовленных выражений (prepared statements).
2.11. Рекурсия (Recursion) — метод программирования, заключающийся в вызове функции из самой себя. В рамках проекта рекурсивные алгоритмы применяются для обработки вложенных структур данных (дерево категорий, файловая система медиа-архива).
2.12. Сниппет (Snippet) — автономный фрагмент программного кода, предназначенный для выполнения конкретной утилитарной задачи (например, парсинг даты, генерация хэша, форматирование валюты) и доступный для повторного использования в различных частях Системы.
2.13. Фронтенд (Frontend / Client-side) — часть Системы, исполняемая в веб-браузере пользователя, обеспечивающая визуализацию данных, обработку событий интерфейса и взаимодействие с серверным API.
2.14. ЧПУ (Clean URL / Human-friendly URL) — технология формирования адресов веб-страниц, основанная на использовании осмысленных слов и иерархических структур вместо набора технических параметров, что способствует улучшению восприятия пользователями и повышению позиций в поисковой выдаче.
2.15. API (Application Programming Interface) — стандартизированный набор программных интерфейсов, реализованный на стороне сервера, позволяющий клиентской части запрашивать данные и инициировать действия через протокол передачи гипертекста (HTTP/HTTPS).
2.16. CMS (Content Management System) — программный интерфейс административного уровня, предоставляющий пользователям инструменты для редактирования, структурирования и публикации информационного наполнения без необходимости прямого вмешательства в программный код.
2.17. JSON (JavaScript Object Notation) — текстовый формат обмена данными, характеризующийся простотой парсинга и компактностью, используемый в Системе для хранения конфигурационных параметров главной страницы и обмена данными через API.
2.18. XSS (Cross-Site Scripting) — тип информационной угрозы, связанный с внедрением деструктивного скрипта в тело веб-страницы. Проект требует внедрения многоуровневой системы фильтрации и экранирования выходных данных.
2.19. CSRF (Cross-Site Request Forgery) — атака, использующая доверенную сессию браузера пользователя для выполнения несанкционированных действий. Требуется обязательное использование одноразовых токенов (nonces) для подтверждения легитимности каждого POST/PUT запроса.
2.20. DOM (Document Object Model) — иерархическая структура элементов веб-страницы, доступная для манипуляций со стороны JavaScript. Оптимизация работы с DOM является ключевым фактором производительности интерфейса.
2.21. CRUD (Create, Read, Update, Delete) — акроним, описывающий полный жизненный цикл данных в рамках административного интерфейса.
2.22. Lazy Loading (Ленивая загрузка) — механизм оптимизации производительности, заключающийся в откладывании инициализации и загрузки тяжелых ресурсов (изображения, видео-плееры) до момента их фактического появления в области видимости пользователя.
2.23. Base64 Encoding — метод кодирования бинарных файлов (например, изображений) в формат ASCII-строки, используемый для прямой вставки графики в CSS-файлы или JSON-объекты с целью сокращения количества HTTP-запросов.
2.24. Middleware (Промежуточное ПО) — слой программного кода, выполняющийся в процессе обработки HTTP-запроса до вызова основного контроллера, предназначенный для проверки авторизации, логирования или трансформации данных.
2.25. Singleton (Одиночка) — фундаментальный паттерн проектирования, гарантирующий создание строго одного экземпляра класса (например, объекта подключения к базе данных) в рамках текущего процесса исполнения.
2.26. WebP / AVIF — передовые форматы сжатия графической информации, обеспечивающие значительное (до 30-50%) снижение объема передаваемых данных по сравнению с классическими форматами JPEG/PNG при идентичном визуальном качестве.
2.27. Чек-лист (Checklist) — формализованный перечень проверок, выполняемый на этапе приемо-сдаточных испытаний для подтверждения соответствия реализации требованиям ТЗ.
2.28. Хотфикс (Hotfix) — экстренное исправление критической ошибки в программном коде, выпускаемое вне рамок планового цикла обновлений.
2.29. Инцидент (Incident) — любое отклонение в работе Системы от штатных показателей, зафиксированное системой мониторинга или пользователем.
2.30. SLA (Service Level Agreement) — соглашение об уровне обслуживания, регламентирующее временные параметры реакции и устранения инцидентов.

3.2.4. Оптимизация пользовательского опыта (UX): Создание интуитивно понятного, быстрого и полностью адаптивного интерфейса, соответствующего современным эстетическим стандартам.

   - Лендинг «ЭТП» (Электронная торговая площадка);

4.1.3. Раздел «Новости»: Динамический архив новостей с поддержкой категорий, тегов и поиска.
4.1.4. Раздел «Услуги»: Каталог направлений деятельности организации с детальными страницами описаний.
4.1.5. Раздел «О компании»: Текстовые страницы с информацией о руководстве, миссии и истории АО.

4.1.8. Навигационная система: Включает главное меню в верхней части страницы, «хлебные крошки» для вложенных страниц и систему контент-меню для лендингов.

4.2. Модель данных и сущности
4.2.1. Сущность «Запись контента» (News/Articles):
   - Уникальный идентификатор (Primary Key).
   - Заголовок (Title) — до 255 символов.
   - Слаг (Slug) — уникальный URL-адрес.
   - Анонс (Excerpt) — краткое описание для превью.
   - Текст (Body) — HTML-содержимое.
   - Миниатюра (Featured Image) — ссылка на медиа-файл.
   - Дата создания и дата модификации (Timestamps).
   - Статус (Черновик / Опубликовано / Запланировано).
   - Автор (связь с сущностью Пользователь).
4.2.2. Сущность «Услуга» (Service):
   - Наименование услуги.
   - Векторная иконка (SVG код).
   - Краткое описание для плиточного отображения.
   - Детальный текст услуги.
   - Цена (опционально, формат Decimal).
   - Ссылка на форму подачи заявки.
4.2.3. Сущность «Настройка» (Setting):
   - Ключ (Key) — уникальная строка.
   - Значение (Value) — текст или сериализованный JSON.
   - Тип данных (Строка, Текст, Файл, Булево).

4.3.5. Гость (Неавторизованный пользователь): Публичный доступ к просмотру материалов сайта.

5. ТРЕБОВАНИЯ К СИСТЕМЕ

5.1. ТРЕБОВАНИЯ К АРХИТЕКТУРЕ И ТЕХНОЛОГИЧЕСКОМУ СТЕКУ
5.1.1. Общие принципы построения
Система строится по принципу разделения ответственности (Separation of Concerns). Основная бизнес-логика и работа с данными сосредоточена на серверной стороне (Backend), в то время как логика представления и взаимодействия с пользователем — на клиентской (Frontend). Обмен данными осуществляется посредством REST-like API в формате JSON.
5.1.2. Требования к серверной части (Backend)
- Язык разработки: PHP версии 8.2 или 8.3 (использование последних стабильных релизов).
- Парадигма: Объектно-ориентированное программирование (OOP) с соблюдением принципов SOLID.
- Работа с базами данных: Только через расширение PDO. Использование драйверов MySQL/MariaDB или SQLite.
- Безопасность кода: Обязательная строгая типизация (strict_types=1), запрет на использование функций `eval()`, `exec()`, `passthru()` без обоснованной технической необходимости.
5.1.3. Требования к клиентской части (Frontend)
- Язык сценариев: Vanilla JavaScript (стандарт ECMAScript 2022+). Поддержка современных браузеров (Chrome, Firefox, Safari, Edge, Яндекс.Браузер). Использование фреймворков (React/Vue/Angular) не допускается для основного функционала с целью минимизации веса страниц.
- Стилизация: Чистый CSS3. Использование CSS-переменных для реализации тем оформления и удобного масштабирования. Запрещено использование CSS-in-JS решений.
- Изображения: Автоматическая конвертация всех загружаемых растровых изображений в формат WebP на стороне сервера.
- Шрифты: Использование современных форматов WOFF2 с механизмом предзагрузки (`preload`).
5.1.4. Требования к системе управления контентом (CMS)
- Реализация: Отдельное веб-приложение или изолированный модуль в рамках основного сайта.
- Безопасность входа: Многофакторная аутентификация (опционально) или ограничение доступа по IP-адресам.
- Визуальный редактор: Интеграция легковесного WYSIWYG-редактора (например, TinyMCE или Quill), настроенного на выдачу чистого семантического HTML.
- Файловый менеджер: Собственная разработка или интеграция решения для управления медиа-библиотекой (загрузка, удаление, переименование, кроппинг).

5.2. ТРЕБОВАНИЯ К ИНФОРМАЦИОННОЙ БЕЗОПАСНОСТИ
5.2.1. Защита от несанкционированного доступа
- Парольная политика: Минимальная длина пароля администратора — 12 символов, обязательное наличие букв в разных регистрах, цифр и спецсимволов.
- Хеширование: Использование встроенных функций PHP `password_hash()` с алгоритмом PASSWORD_ARGON2ID.
- Защита от перебора (Brute-force): Блокировка учетной записи или IP-адреса после 5 неудачных попыток входа на 15 минут.
5.2.2. Защита данных и предотвращение атак
- SQL Injection: Категорический запрет на конкатенацию строк при формировании SQL-запросов. Использование только подготовленных выражений (Prepared Statements).
- XSS Protection: Все данные, выводимые на страницу, должны проходить через функцию `htmlspecialchars()` или аналогичные механизмы шаблонизатора.
- CSRF Protection: Каждая форма, инициирующая изменение данных, должна содержать уникальный скрытый токен, проверяемый на стороне сервера.
- File Upload Security: Загружаемые файлы должны проверяться на соответствие расширению и MIME-типу. Имена файлов должны генерироваться заново для предотвращения атак типа Directory Traversal.
5.2.3. Безопасность сетевого взаимодействия
- Протокол: Обязательное использование TLS версии 1.3.
- Заголовки: Настройка HSTS (HTTP Strict Transport Security) для предотвращения атак типа SSL Stripping.

5.3. ТРЕБОВАНИЯ К SEO И ОПТИМИЗАЦИИ ДЛЯ ПОИСКОВЫХ СИСТЕМ
5.3.1. Управление мета-данными
Система должна предоставлять интерфейс для редактирования следующих параметров каждой страницы:
- Заголовок окна браузера (Title tag).
- Мета-описание (Description tag).
- Мета-ключевые слова (Keywords tag) — опционально.
- Тег Canonical для предотвращения дублирования контента.
- Настройки индексации (Robots meta: noindex, nofollow).
- ЧПУ (человекопонятный URL) для каждой страницы.
Система должна содержать отдельный интерфейс (список всех страниц) в административной панели для оперативного управления вышеуказанными полями без перехода в режим редактирования каждой отдельной страницы.
5.3.2. Техническая оптимизация
- Иерархия заголовков: Автоматическая проверка наличия одного тега H1 на странице и логической последовательности H2-H6.
- Атрибуты изображений: Обязательное поле «Alt-текст» в медиа-библиотеке для каждого изображения.
- Скорость загрузки: Использование Gzip/Brotli сжатия текстовых ресурсов на уровне веб-сервера.
- Микроразметки: Автоматическое внедрение разметки Open Graph для корректного отображения ссылок в социальных сетях и мессенджерах.

5.4. ТРЕБОВАНИЯ К НАДЕЖНОСТИ И ДОСТУПНОСТИ
5.4.1. Режим функционирования
Система должна обеспечивать непрерывный режим работы (24 часа в сутки, 7 дней в неделю). Допустимое время простоя для проведения регламентных работ не должно превышать 4 часов в месяц в ночное время (с 01:00 по 05:00 МСК).
5.4.2. Резервное копирование и восстановление
- База данных: Ежедневное создание полной копии БД (dump) с хранением на отдельном физическом или облачном носителе. Глубина хранения — 30 последних копий.
- Файловая система: Еженедельное резервное копирование медиа-файлов и исходного кода.
- Время восстановления: В случае критического сбоя серверного оборудования, время развертывания Системы из последней резервной копии не должно превышать 60 минут.
5.4.3. Обработка ошибок
Система должна иметь механизм перехвата критических ошибок (Exceptions) и их логирования в специализированный файл или базу данных. Пользователю должна отображаться стилизованная страница ошибки (404, 500) без раскрытия технических подробностей работы сервера (стек вызовов, пути к файлам).

ТЕХНИЧЕСКИЕ ТРЕБОВАНИЯ К ЗАГРУЖАЕМЫМ МЕДИА-МАТЕРИАЛАМ

1.1. Требования к графическим файлам (Изображения)
Все графические материалы, загружаемые через административную панель, должны соответствовать следующим параметрам:
- Допустимые форматы: WebP (приоритетный), JPG, PNG, SVG (только для иконок и логотипов).
- Максимальное разрешение: 1920x1080 пикселей для полноэкранных баннеров; 800x600 пикселей для новостных лент.
- Оптимизация: Все растровые изображения должны проходить через алгоритм сжатия без видимой потери качества (Lossy compression) до объема не более 300 КБ для одной единицы.

1.2. Требования к векторной графике (SVG)
- Код SVG должен быть очищен от метаданных графических редакторов (Adobe Illustrator, Figma) с помощью инструмента SVGO.
- Запрещено использование внедренных растровых изображений (Base64) внутри SVG-файлов.
- Все шрифты в SVG должны быть преобразованы в кривые (Outlines).

1.3. Требования к документам (Документация для скачивания)
- Допустимые форматы: PDF, DOCX, XLSX.
- Рекомендуемый размер файла: До 10 МБ. В случае превышения лимита рекомендуется разбивка на части или архивация (ZIP).
- Имена файлов: Использование латиницы, цифр и знака подчеркивания. Запрещено использование кириллицы и пробелов в названиях файлов (например, `regulations_2026_v1.pdf`).

- Рекомендуется использование внешних видео-хостингов (YouTube, RuTube, VK Video) с вставкой через `iframe` или специализированные API-плееры.
