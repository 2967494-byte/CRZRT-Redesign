## 1. ВВЕДЕНИЕ

### 1.1. Назначение системы администрирования

Административная подсистема (CMS) представляет собой закрытый веб-интерфейс, доступный по адресу `admin.html`, предназначенный для централизованного управления всеми информационными разделами портала без необходимости модификации программного кода.

В рамках настоящего Руководства под **администратором** понимается уполномоченный сотрудник Заказчика, обладающий учётной записью с ролью `superadmin` или `admin` в таблице `users` базы данных, а также (при необходимости) доступом к серверу по SSH/FTP и панели управления хостингом (ISPmanager).

### 1.2. Состав административного комплекса

| Компонент | URL / путь | Назначение |
|-----------|------------|------------|
| Основная CMS | `/new_design/admin.html` | Редактирование всех разделов |
| Календарь курсов | `/new_design/admin-obuchenie-courses.html` | Реестр курсов обучения |
| API авторизации | `/new_design/api/auth.php` | Сессии, login/logout |
| API настроек | `/new_design/api/settings.php` | Хранение JSON-контента |
| Установщик | `/new_design/api/install.php` | Инициализация БД |
| Конфигурация БД | `/new_design/api/db.php` | Параметры PDO |

### 1.3. Роли и зоны ответственности

В соответствии с матрицей ролей ТЗ (п. 4.3):

| Роль | Зона ответственности |
|------|---------------------|
| **Супер-администратор** | Пользователи, БД, бэкапы, Bitrix webhook, deploy |
| **Администратор/Редактор** | Контент страниц, медиа, новости (без доступа к серверу) |
| **Контент-менеджер** | См. отдельное Руководство контент-менеджера |

## 2. ТРЕБОВАНИЯ К АДМИНИСТРАТОРУ

### 2.1. Квалификационные требования

Администратор системы должен обладать базовыми компетенциями в области:

- администрирования Linux-серверов (SSH, права файлов, cron);
- работы с MySQL/MariaDB (дамп, восстановление, простые запросы);
- понимания HTTP, HTTPS, DNS, SSL-сертификатов;
- основ веб-безопасности (сессии, пароли, резервные копии).

### 2.2. Технические средства

- Современный браузер: Chrome 120+, Firefox 120+, Edge 120+, Яндекс.Браузер;
- Разрешение экрана не менее 1280×720 (рекомендуется 1920×1080);
- Стабильное подключение к сети Интернет;
- Для серверных операций: SSH-клиент (PuTTY, OpenSSH), FTP/SFTP-клиент (FileZilla).

## 3. ДОСТУП К СИСТЕМЕ

### 3.1. Вход в административную панель

**Порядок действий:**

1. Откройте в браузере адрес: `https://zakupki.tatar/new_design/admin.html`
2. При отсутствии активной сессии отобразится модальное окно «Вход в систему»
3. Введите **E-mail** и **Пароль** учётной записи администратора
4. Нажмите кнопку **«Войти»**
5. При успешной аутентификации модальное окно закроется, станет доступна боковая панель навигации

**Техническая реализация:** запрос `POST api/auth.php?action=login` с телом `{email, password}`. При успехе сервер устанавливает cookie `PHPSESSID` и записывает в сессию `user_id`, `user_role`, `user_name`.

### 3.2. Первичная учётная запись

После выполнения скрипта `api/install.php` создаётся пользователь:

- **E-mail:** `admin@crzrt.ru`
- **Роль:** `superadmin`
- **Пароль:** задаётся при install (по умолчанию — временный; **обязательно смените после первого входа**)

> **ВАЖНО (ТЗ п. 5.2.1):** Пароль администратора должен содержать не менее 12 символов, включая буквы верхнего и нижнего регистра, цифры и спецсимволы.

### 3.3. Выход из системы

Для завершения сессии используйте кнопку выхода в интерфейсе CMS (при наличии) или выполните запрос `GET api/auth.php?action=logout`. Рекомендуется всегда завершать сессию при работе на общедоступных компьютерах.

### 3.4. Ограничение доступа по IP (рекомендация ТЗ п. 5.1.4)

Для повышения безопасности рекомендуется настроить на уровне Nginx/Apache ограничение доступа к `admin.html` и каталогу `api/` (кроме публичных GET settings и bitrix-lead-enroll) по whitelist IP-адресов офиса Заказчика.

**Пример для Nginx:**

```nginx
location ~ ^/new_design/(admin\.html|admin-.*\.html|api/(auth|upload|settings)) {
    allow 185.x.x.x/32;   # офис Заказчика
    deny all;
}
```

### 3.5. Переключение темы оформления

В правом верхнем углу CMS расположена кнопка переключения светлой/тёмной темы (`theme-toggle-admin`). Выбор сохраняется в `localStorage` браузера под ключом `theme`.

### 4.3. Кнопка «Сохранить изменения»

Глобальная кнопка сохранения (`btnSaveAll`) расположена в верхней части рабочей области. При нажатии:

1. Определяется активный раздел (`currentTarget`)
2. Собирается JSON-объект текущих данных формы
3. Выполняется `POST api/settings.php` с соответствующим `key`
4. При успехе — дублирование в `localStorage` и уведомление пользователя

**Важно:** до нажатия «Сохранить» изменения существуют только в оперативной памяти браузера и будут утрачены при перезагрузке страницы.

### 4.4. Отдельная страница «Календарь курсов»

Полноценный реестр курсов вынесен на `admin-obuchenie-courses.html` (ссылка «Календарь курсов» в разделе Обучение). Функции:

- табличный просмотр всех курсов;
- добавление / редактирование / удаление;
- привязка форм Bitrix24 (ФЛ/ЮЛ);
- автоматическое создание лида в CRM при новом курсе;
- WYSIWYG-описание курса.

### 4.5. Элементы управления медиа

Во всех разделах CMS единообразно реализованы:

- **Кнопка загрузки изображения** — открывает диалог выбора файла → Cropper.js → отправка на `api/upload.php`;
- **Поле URL** — ручной ввод или автозаполнение после загрузки;
- **Превью** — миниатюра текущего изображения;
- **Слот (`slot`)** — идентификатор назначения файла для именования на сервере.

## 5. УПРАВЛЕНИЕ ПОЛЬЗОВАТЕЛЯМИ

### 5.1. Два уровня управления пользователями

В системе реализованы **два взаимодополняющих механизма**:

#### 5.1.1. Серверные учётные записи (MySQL)

**Таблица:** `users`  
**Аутентификация:** `api/auth.php`  
**Назначение:** реальный вход в CMS, защита API записи

Управление:

- создание первого пользователя — `api/install.php`;
- добавление пользователей — SQL INSERT или расширение `auth.php` (при развитии системы);
- смена пароля — SQL UPDATE `password_hash` с `password_hash('новый_пароль', PASSWORD_DEFAULT)`.

#### 5.1.2. Клиентский реестр прав (localStorage)

**Ключ:** `crzrt_users` в `localStorage`  
**Управление:** раздел «Пользователи» в CMS  
**Назначение:** гранулярные права на разделы CMS

### 5.2. Интерфейс раздела «Пользователи»

**Путь:** CMS → боковое меню → **Пользователи**

**Элементы интерфейса:**

| Элемент | ID / класс | Описание |
|---------|------------|----------|
| Таблица пользователей | `usersListContent` | Список e-mail, имени, прав |
| Кнопка «Добавить» | `btnAddUser` | Открытие формы нового пользователя |
| Кнопка «Редактировать» | в строке таблицы | Изменение имени и прав |
| Кнопка «Удалить» | в строке таблицы | Удаление из `crzrt_users` |

### 5.3. Матрица разрешений (permissions)

| Permission | Описание | Доступные разделы |
|------------|----------|-------------------|
| `superuser` | Полный доступ | Все разделы CMS |
| `users` | Управление кадрами | Только «Пользователи» |
| *(индивидуальные)* | Настраиваются чекбоксами | Выбранные лендинги |

### 5.4. Процедура добавления нового администратора

**Полная процедура (рекомендуемая):**

1. **Сервер:** создайте запись в таблице `users`:

```sql
INSERT INTO users (name, email, password_hash, role)
VALUES (
  'Иванов Иван',
  'ivanov@crzrt.ru',
  '$argon2id$...',  -- сгенерировать через password_hash()
  'admin'
);
```

2. **CMS:** раздел «Пользователи» → «Добавить» → указать тот же e-mail, имя, права
3. **Проверка:** выйти из CMS, войти под новой учётной записью
4. **Документирование:** зафиксировать в журнале администрирования

### 5.5. Процедура блокировки пользователя

1. Удалить или изменить запись в `crzrt_users` (клиентские права)
2. Удалить или деактивировать запись в таблице `users` (серверный вход)
3. При подозрении на компрометацию — сменить `password_hash` и инвалидировать сессии (перезапуск PHP-FPM)

### 5.6. Политика паролей (ТЗ п. 5.2.1)

| Требование | Значение |
|------------|----------|
| Минимальная длина | 12 символов |
| Сложность | A-Z, a-z, 0-9, спецсимволы |
| Алгоритм хеширования | ARGON2ID (рекомендуется) или bcrypt |
| Блокировка при переборе | 5 попыток / 15 минут (рекомендуется настроить) |

## 6. НАСТРОЙКИ СИСТЕМЫ

### 6.1. Раздел «Настройки» в CMS

Содержит глобальные параметры сайта, не привязанные к конкретному лендингу:

- контактные данные организации;
- параметры SEO (при реализации в интерфейсе);
- служебные флаги.

### 6.2. Раздел «Контакты и реквизиты»

Позволяет редактировать:

- юридический и фактический адрес;
- телефоны, e-mail;
- реквизиты (ИНН, КПП, ОГРН);
- ссылки на карты и мессенджеры.

Данные сохраняются под ключом `crzrt_contacts`.

### 6.3. Конфигурация базы данных

**Файл:** `api/db.php`

Содержит параметры PDO-подключения. При смене хостинга администратор обязан:

1. Создать новую БД и пользователя MySQL
2. Обновить `$host`, `$db`, `$user`, `$pass` в `db.php`
3. Импортировать дамп (см. раздел 8)
4. Проверить работу `api/settings.php`

> **Безопасность:** файл `db.php` не должен быть доступен для скачивания извне. Убедитесь, что веб-сервер не отдаёт `.php` как статику.

### 6.4. Файл `.htaccess`

Содержит правила Apache (при использовании). Проверьте наличие:

- запрета листинга каталогов;
- редиректа на HTTPS;
- защиты служебных файлов.

## 8. РЕЗЕРВНОЕ КОПИРОВАНИЕ И ВОССТАНОВЛЕНИЕ

### 8.1. Регламент (ТЗ п. 5.4.2)

| Объект | Периодичность | Глубина хранения | Место хранения |
|--------|---------------|------------------|----------------|
| База данных MySQL | **Ежедневно** | 30 последних копий | Отдельный носитель / облако |
| Медиа `uploads/` | **Еженедельно** | 4–8 копий | Отдельный носитель |
| Исходный код | **Еженедельно** | По политике git | Репозиторий + архив |

**Целевое время восстановления (RTO):** не более 60 минут.

### 8.2. Создание дампа базы данных

**Вручную:**

```bash
mysqldump -u DB_USER -p DB_NAME > backup_$(date +%Y%m%d).sql
gzip backup_$(date +%Y%m%d).sql
```

**Автоматизация (cron, ежедневно в 03:00):**

```cron
0 3 * * * mysqldump -u USER -p'PASS' DB_NAME | gzip > /backups/db/crzrt_$(date +\%Y\%m\%d).sql.gz
```

**Ротация (хранить 30 копий):**

```bash
find /backups/db/ -name "crzrt_*.sql.gz" -mtime +30 -delete
```

### 8.3. Резервное копирование файлов

```bash
tar -czf /backups/files/uploads_$(date +%Y%m%d).tar.gz /path/to/new_design/uploads/
tar -czf /backups/code/new_design_$(date +%Y%m%d).tar.gz /path/to/new_design/ \
  --exclude='uploads' --exclude='api/bitrix-config.local.php'
```

> **Важно:** файл `api/bitrix-config.local.php` содержит секрет webhook Bitrix24. Храните его отдельно в защищённом хранилище паролей (не в публичном git).

### 8.4. Восстановление из резервной копии

**Порядок действий при критическом сбое:**

1. **Оценка инцидента** — определить масштаб (БД, файлы, полный сбой)
2. **Восстановление БД:**

```bash
gunzip -c backup_20260703.sql.gz | mysql -u USER -p DB_NAME
```

3. **Восстановление uploads:**

```bash
tar -xzf uploads_20260703.tar.gz -C /
```

4. **Восстановление кода** (при необходимости):

```bash
tar -xzf new_design_20260703.tar.gz -C /var/www/
```

## 9. ИНТЕГРАЦИЯ BITRIX24

### 9.1. Назначение

Синхронизация лидов CRM при:

- создании курса в `admin-obuchenie-courses.html`;
- отправке заявки «Записаться» на `obuchenie.html`.

### 9.2. Настройка webhook

1. В Bitrix24: Приложения → Вебхуки → Создать → права `crm` → `crm.lead.add`
2. Скопировать URL webhook
3. Создать на сервере `api/bitrix-config.local.php`:

```php
<?php
return [
    'webhook_lead_add' => 'https://PORTAL.bitrix24.ru/rest/USER/KEY/crm.lead.add.json',
];
```

4. Проверить:

```bash
curl -X POST "https://zakupki.tatar/new_design/api/bitrix-lead-enroll.php" \
  -H "Content-Type: application/json" \
  -d '{"name":"Test","phone":"+79000000000","courseTitle":"Проверка"}'
```

Ожидаемый ответ: `{"success":true,"leadId":...}`

### 9.3. Диагностика проблем

| Симптом | Причина | Решение |
|---------|---------|---------|
| «Webhook не настроен» | Нет `bitrix-config.local.php` | Создать файл на сервере |
| HTTP 403 при курсе | Нет сессии в admin-courses | Войти в админку заново |
| Лид не в CRM, но success | Неверный портал/права webhook | Пересоздать webhook с crm |
| Старые курсы без лида | Созданы до интеграции | Пересохранить курс в админке |
